INTRODUCTION

SWECHAT project is a web based chat application. The application features a server that forms the backbone of the chat system and also features clients which communicate with other chat clients in a peer-to-peer messaging mode. To test the complete functionality, security, usability and durability of the product, we will use different testing methods, cases and software which are described in this document.


GOALS AND OBJECTIVES

The overall goals of the test processes are intended to prove that the functions of the product work as specified in RFC documents. Additionally, they are meant to uncover possible bugs.


STATEMENT OF SCOPE

The scope of the testing would be all functional and non-functional activity within the product.


MAJOR CONSTRAINTS

A primary constraint for testing would be the;

  • Web server which server is hosted on.
  • Client application should be tested on different machine from same or different network with server.
  • Wireshark software to analyze TCP packets.

TEST PLAN

SOFTWARE TO BE TESTED

The product itself is to be tested as explicitly as possible. SWECHAT consists of two different main parts: Server and Client. Their interactions and the flow of information between Client to Server, Server to Client and Client to Client will be fully tested.

This section will outline the test plan that will be implemented in order to properly test SWECHAT. These tests will be done using the concepts of smoke testing, which will allow us to test larger piece of the product at a time depending on how the individual function relate to each other.

Installation Testing

Testing the installation of both client and server applications. Checking if both applications can be installed as defined in “User Manual”.

Unit Testing

The unit testing strategy involves breaking the system down into individual fields in which we will refer to as functions. All functions defined in RFC involving client, server, client-client and client-server will be tested.

Compatibility Testing

Application will be tested on different OS such as Windows and Linux (Did we agree on just using one OS?)

Software Performance Testing

Performance and durability of application will be tested. Application should respond to 10 requests in one second as normally which means it should support at least 10 clients online.

Validation Testing

Upon completing the tests that are preformed in installation testing and unit testing, that validity of the units tested will need to be confirmed. This means that output and functionality of the units will need to be compared to the language that was set forth in the software specification document. Validation testing will make sure the final product has the correct functionality throughout.


TESTING RESOURCE

The resources to be used for testing include web server, client applications and Wireshark software.


TEST RECORD KEEPING

For recording our test results, the proposed idea is to create table keeping track of when our functions pass and fail their specific tests. This method will allow us to quickly fill in the results of tests and refer to them later.


TESTING METRICS

Both applications will be tested to meet the quality and technical expectations proposed in the requirements specification. Usability, efficiency, reliability and completeness are a few important metrics that need to be accounted for during testing activity.

  • Usability: this will take into account a users' ability to navigate and use the software with ease. Although this can be applied after software builds, this metric will be taken into higher consideration at the final development stages of the product.
  • Reliability: this metric will verify that functions will have consistent results after each operation.
  • Completeness: this metric will be used to test functions and the software as a whole, rating their functionality to the final versions specified in requirements.

TEST SCHEDULE

First installation testing will be completed. Then unit testing will be done. As each unit is tested it will be integrated with previously tested units in a top-down fashion. As new things are integrated, tests will be done to ensure that the product is working properly and there are not unforeseen side effects. During this process, the units will be tested against the specifications to ensure they meet the requirements. After each unit tested, we will check security, performance, and durability of the software.


NOTES

If in any test case an error is expected as a response, then an error with a message field is also accepted. If the test case was an injection case, then the server can choose to respond with any ERR packet, empty response, or no response for 30 seconds.

Unless otherwise stated, test users have no contacts, offline messages, or requests.

Although the sequence number, IP address, and date fields in test data are not explicitly specified, during the actual tests these values will be checked for correctness.

When an error happens, the client is expected to display an error message if the user needs to be notified, or do not display anything if the error needs to be handled silently

If a message is injected at any point in the test cases, it is expected that the client will not allow the user to perform that action at that point of the test (or ever) via the user interface

The Client UI might implement denying a request as ignoring the request while implementing the same functionality.
